Flight Anomaly Live Learning Monitor: Detailed Project Explanation for Academic Use

1. Project Title
Flight Anomaly Live Learning Monitor using Flask, Online Machine Learning, and Real-Time Flight Telemetry Simulation

2. Project Overview
This project is a full-stack simulation and monitoring system designed to demonstrate how live flight operations can be observed, analyzed, and assessed for anomalies in real time. The system imitates an aircraft travelling along a planned route while continuously generating synthetic operational data related to flight path behavior, engine health, and airspace conditions. As new observations arrive, the application updates online anomaly detection models and visualizes the evolving state through an interactive web dashboard.

The project combines three important ideas:
1. Real-time systems monitoring
2. Incremental or online learning for anomaly detection
3. Human-readable dashboard-based decision support

From an academic perspective, the project can be positioned as a proof-of-concept intelligent aviation monitoring framework. It does not depend on live aircraft data feeds; instead, it uses seeded CSV inputs and a simulation engine to recreate realistic operational scenarios. This makes the system portable, reproducible, and suitable for demonstrations, coursework, and experimental comparison of anomaly detection strategies.

3. Main Objective
The primary objective of the project is to build a monitoring platform that can:
- Track the progress of a simulated flight from origin to destination
- Detect deviations between the actual route and the planned route
- Monitor engine and subsystem telemetry for abnormal operating conditions
- Assess airspace conditions and disruption-related risk
- Compare a continuously updating live-learning model with static baseline models
- Present all outputs in a clear dashboard for interpretation

In simple terms, the project shows how artificial intelligence can assist in flight monitoring by continuously learning from incoming data and highlighting patterns that may indicate unsafe or unusual behavior.

4. Problem Statement
In real aviation and other safety-critical systems, anomalies may emerge gradually rather than as isolated events. Traditional offline models are trained once and may become less effective when the operating environment changes. This project addresses that challenge by using online learning methods that update as each new simulated observation arrives. The system therefore demonstrates the practical advantage of adaptive monitoring over static modeling.

5. What the Project Does
The application performs the following workflow:

Step 1:
It loads initial seed data from local CSV files. These files define:
- A planned flight route using latitude and longitude waypoints
- A baseline engine telemetry snapshot
- Initial airspace news or disruption messages

Step 2:
It starts a simulation engine that updates once per second in a background thread.

Step 3:
At every simulation step, the engine:
- Moves the aircraft toward the next route waypoint
- Adds controlled random variation to represent route deviation and weather effects
- Synthesizes engine telemetry values such as temperature, vibration, oil pressure, hydraulic pressure, fuel flow, and avionics bus voltage
- Generates airspace status values such as open, restricted, and shutdown corridors
- Produces anomaly labels using rule-based thresholds
- Feeds features and labels into online anomaly detection models

Step 4:
It stores the latest state and history so that the frontend can display:
- Planned route versus actual path
- Path anomalies
- Engine health metrics and anomaly score
- Airspace status, bulletin messages, and anomaly score
- Comparative performance of live-learning versus static models

Step 5:
The web frontend requests updates through REST API endpoints and renders the new information on the dashboard.

6. Project Structure
The repository is organized into backend logic, frontend presentation, seed data, and the application entry point.

6.1 Entry Point
File: run.py

Purpose:
This file launches the Flask application.

Key role:
- Imports the Flask app factory from the app package
- Creates the app instance
- Runs the development server on port 5000

6.2 Application Initialization
File: app/__init__.py

Purpose:
This file creates the Flask application and exposes the HTTP routes.

Important behavior:
- Instantiates a single global FlightSimulationEngine object
- Starts a background daemon thread
- The background thread calls the simulator step function once every second
- Defines the frontend routes:
  - "/" for the live monitoring dashboard
  - "/compare-models" for the model comparison dashboard
- Defines the API routes:
  - GET /api/state
  - GET /api/compare
  - POST /api/control
  - POST /api/reset

This file acts as the bridge between the simulator and the user interface.

6.3 Data Loading Layer
File: app/data_loader.py

Purpose:
This module loads the seed CSV data required to initialize the system.

Functions:
- load_route()
  Reads the route CSV and converts each row into latitude-longitude coordinates.
- load_engine_baseline()
  Reads the first row of engine baseline telemetry and converts values to numeric form.
- load_airspace_news()
  Reads seeded airspace bulletin data and severity scores.

Why it matters:
This modular loading layer makes the project configurable. The CSV files can be replaced by new datasets without changing the simulation logic.

6.4 Modeling Layer
File: app/models.py

Purpose:
This module defines the anomaly detection models.

It includes two major components:

6.4.1 OnlineBinaryAnomalyModel
This is an online binary anomaly detector.

How it works:
- If scikit-learn is available, it uses:
  - StandardScaler for online feature normalization
  - SGDClassifier with log-loss for incremental probabilistic classification
- The model receives one sample at a time through partial_fit()
- It returns an anomaly probability score between 0 and 1
- If scikit-learn is not available, the code falls back to a statistical z-score style mechanism using a running center and variance

Why this is important:
This component demonstrates incremental learning. The model does not require retraining on the full dataset. Instead, it updates continuously as new samples arrive.

6.4.2 TinyAutoencoder
This is a lightweight neural-network-inspired model that behaves like a tiny online autoencoder.

How it works:
- Input data is compressed into a hidden representation using tanh activation
- The hidden representation is reconstructed back into the original feature space
- Mean squared reconstruction error is calculated
- A single-sample stochastic gradient descent update is applied to the weights
- The reconstruction error is normalized with an exponential moving average
- The final output is converted into an anomaly score

Why this is important:
The autoencoder-like model provides a deep-learning-style signal. It is especially useful for detecting unusual engine telemetry patterns that differ from normal behavior even when explicit labels are limited.

7. Core Simulation Engine
File: app/simulator.py

This is the central logic of the project.

7.1 Main Role
The FlightSimulationEngine class simulates the flight process, creates telemetry data, updates models, stores histories, and produces API-ready output.

7.2 Internal Components
The simulation engine includes:
- A thread lock for safe shared access
- A seeded random number generator for reproducibility
- Control parameters that allow users to change scenario severity
- One-shot injection parameters for sudden anomaly events
- Route storage and current position tracking
- Path, engine, and airspace anomaly models
- An autoencoder for engine reconstruction error
- History buffers for route visualization and model comparison
- Static baseline models for performance comparison

7.3 Control Variables
The user can dynamically tune four high-level controls:
- route_deviation
- engine_stress
- weather
- airspace_risk

Each value ranges from 0 to 1 and changes the behavior of the simulator.

Interpretation:
- Higher route deviation increases path instability
- Higher engine stress increases temperature and vibration while reducing some health metrics
- Higher weather severity degrades speed and operational conditions
- Higher airspace risk increases restrictions and corridor shutdown probability

7.4 One-Shot Event Injections
The interface also supports event injections:
- engine_temp_delta
- vibration_delta
- force_airspace_shutdown

These simulate sudden abnormal incidents. After each step, the numeric injections decay by multiplying by 0.85, which produces a temporary spike effect instead of a permanent jump.

7.5 Route Construction
The engine first attempts to load the flight route from the CSV seed file. If the route file is unavailable, the code falls back to a predefined route. In the current repository, the loaded route corresponds to an Indian corridor and includes waypoints from Delhi to Mumbai.

7.6 Flight Motion Logic
At every step:
- The engine identifies the next target waypoint
- It moves 25 percent toward that waypoint
- It adds random noise controlled by route deviation and weather
- If the aircraft comes sufficiently close to the target waypoint, the simulator advances to the next waypoint

This produces a smooth but imperfect trajectory that appears realistic for monitoring and anomaly analysis.

7.7 Path Feature Engineering
The path anomaly model uses five features:
- Distance from planned route in kilometers
- Heading error in degrees
- Speed in knots
- Weather severity
- Route deviation

The path anomaly label is generated as:
- anomaly = 1 if distance_from_plan_km > 7 or heading_error_deg > 20
- otherwise anomaly = 0

This creates a supervised online classification setup in which the model continuously learns from synthetic labeled observations.

7.8 Engine Telemetry Synthesis
The simulator generates six core engine/subsystem features:
- engine_temp_c
- vibration_g
- oil_pressure_psi
- hydraulic_psi
- fuel_flow_kg_h
- avionics_bus_v

These variables are driven by stress, weather, deviation, random noise, and event injections.

Examples:
- Higher engine stress increases temperature, vibration, and fuel flow
- Higher engine stress decreases oil pressure and hydraulic pressure
- Event injections can push temperature or vibration higher instantly

The engine anomaly model uses eight features:
- The six telemetry variables above
- engine stress
- weather severity

The engine anomaly label becomes 1 if any critical threshold is crossed, including:
- engine_temp_c > 780
- vibration_g > 3.9
- oil_pressure_psi < 42
- hydraulic_psi < 2700

The final engine anomaly score is a weighted fusion:
- 65 percent online ML score
- 35 percent autoencoder reconstruction score

This hybrid design is a major academic strength of the project because it combines label-driven online learning with unsupervised reconstruction-based abnormality detection.

7.9 Airspace Condition Simulation
The simulator also models external operational context.

It computes:
- open_corridors
- restricted_corridors
- shutdown_corridors
- latest_news bulletins

These are derived from:
- airspace risk control
- weather severity
- random probabilistic events
- optional forced shutdown injection

The airspace feature vector contains:
- open corridors
- restricted corridors
- shutdown corridors
- average news severity
- airspace risk control value

The airspace label becomes anomalous when:
- a shutdown corridor exists, or
- shutdown-related feature intensity exceeds threshold

This demonstrates that anomaly detection in aviation is not limited to mechanical behavior; operational environment intelligence is also essential.

8. Static Model Comparison
One major contribution of this project is the explicit comparison between adaptive and non-adaptive approaches.

8.1 Static Baseline Models
During reset, the system builds two baseline classifiers:
- Random Forest
- XGBoost if available, otherwise Gradient Boosting fallback

Training data for these static models is not collected from external sources. Instead, the project programmatically generates synthetic training samples using the same path-related factors:
- distance
- heading
- speed
- weather
- deviation

Labels are assigned using the same path anomaly rules as the live model. This creates a controlled baseline environment for comparison.

8.2 Comparison Metrics
For every time step, the simulator records:
- True label
- Live-learning score and prediction
- Random Forest score and prediction
- XGBoost score and prediction

The compare endpoint computes:
- Accuracy
- Precision
- Recall
- F1-score

It also reports the performance gap:
- accuracy_live_vs_rf
- accuracy_live_vs_xgb
- f1_live_vs_rf
- f1_live_vs_xgb

This allows a viewer to examine whether online learning is keeping pace with evolving conditions more effectively than static models.

9. Frontend and Visualization

9.1 Main Dashboard
Files:
- app/templates/index.html
- app/static/js/app.js
- app/static/css/style.css

The dashboard has three main monitoring windows:
- Left panel:
  Planned route, actual path, current location, and route anomaly markers on a Leaflet map
- Right top panel:

  Airspace corridor status, latest disruption bulletins, and anomaly badge

Below the main dashboard there is a utility control window where the user can:

- Inject temporary anomalies
- Reset the simulation
- Move to the model comparison page

9.2 Model Comparison Dashboard
Files:
- app/templates/compare_models.html
- app/static/js/compare_models.js

This dashboard visualizes three models side by side:
- Live Learning
- Static Random Forest
- Static XGBoost

For each model, it shows:
- The path on a map
- Highlighted anomaly points
- A chart of running accuracy and F1-score
- Summary metrics

This page strengthens the academic value of the project by showing not only system monitoring but also experimental model evaluation.

10. API Design

10.1 GET /api/state
Returns the current simulator state, including:
- timestamp
- controls
- flight progress and positions
- path metrics
- engine metrics and anomaly scores
- airspace metrics and news

10.2 GET /api/compare
Returns data needed for the comparison page:
- planned route
- current progress
- per-model path history
- per-model metric series
- performance summaries
- advantage gap statistics

10.3 POST /api/control
Accepts JSON input for:
- controls
- injections

It updates the simulator configuration without restarting the server.

10.4 POST /api/reset
Resets the entire simulation state, reloads seed data, rebuilds baseline models, and starts the scenario from the beginning.

13. How the Project Achieves Its Goal
The project achieves its objective by integrating simulation, online learning, and dashboard visualization into one pipeline.

Achievement chain:
1. Seed data initializes realistic route and monitoring context
2. The simulator creates continuous flight, engine, and airspace observations
3. Online models learn incrementally from those observations
4. Anomaly scores are generated at each step
5. API endpoints expose the updated state
6. The frontend visualizes the outputs for a human operator
7. Static models are compared against the live-learning model to demonstrate adaptive benefits

14. Academic Contributions and Strengths
This project has several strengths that are useful in an academic report.

14.1 Demonstration of Online Learning
Many student projects use static datasets and offline training. This project instead demonstrates incremental learning using partial_fit, which is more aligned with real-time monitoring environments.

14.2 Multi-Domain Monitoring
The project does not focus on just one feature family. It integrates:
- Flight path monitoring
- Engine health monitoring
- Airspace intelligence

This gives the system broader operational relevance.

14.3 Hybrid Intelligence
The engine anomaly score combines:
- A supervised online classifier
- A small unsupervised autoencoder-style learner

This is a strong architectural choice because real anomalies may be captured by both rule-informed labels and reconstruction error patterns.

14.4 Human-Centered Visualization
The dashboard design supports interpretation by human users. Instead of only producing numerical outputs, it explains state through maps, badges, metrics, and time-evolving charts.

14.5 Experimental Comparison
The model comparison page introduces a research-like evaluation perspective by comparing live adaptive learning with fixed tree-based baselines.

15. Practical Output of the Project
When the project is running successfully, the user observes:
- An aircraft progressing along a route
- Actual path drift under different conditions
- Route anomaly marking when deviations become severe
- Engine metric changes under stress or injection events

Thus, the final output is not merely a classification result. It is a complete monitoring environment with both operational and analytical value.

16. Limitations
For academic honesty, the following limitations should also be acknowledged:

16.1 Simulated Data
The system uses synthetic and seeded data rather than live aircraft telemetry or official aviation feeds.

16.2 Simplified Physics
Flight movement and subsystem behavior are generated through heuristic mathematical relationships, not full aerodynamic or mechanical models.

16.3 Rule-Based Labels
Ground-truth anomaly labels are generated by threshold rules rather than expert-annotated operational logs.

16.4 No Persistent Storage
The project stores state in memory only. Restarting the application resets the run.

16.5 Limited Security and Production Hardening
The Flask application is suitable for demonstration and academic use, not direct production deployment.

In conclusion, this project successfully demonstrates how a live intelligent monitoring system can be built using lightweight, reproducible components. It simulates a realistic flight route, creates evolving operational data, applies online anomaly detection, compares adaptive and static models, and presents the full state through a web dashboard.

The project achieves an effective combination of:
- real-time simulation
- continuous learning
- multi-source anomaly analysis
- interpretable visualization

For academic purposes, the project is valuable because it clearly shows the relationship between data generation, feature engineering, machine learning, deep-learning-inspired scoring, system architecture, and decision-support presentation within a single coherent application.
